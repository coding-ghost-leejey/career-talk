## 3. UI/UX, 결국 무슨 뜻일까?
카페에 비유해보면 훨씬 쉬워요.

**UI — User Interface**: 눈에 보이는 모든 것
카페의 인테리어, 메뉴판 디자인, 테이블 배치, 조명 색깔. 앱에서는 버튼, 색, 폰트, 아이콘, 화면 레이아웃.

**UX — User Experience**: 그 안에서 느끼는 경험 전체
주문은 편했는지, 자리는 찾기 쉬웠는지, 다시 오고 싶은지. 앱에서는 원하는 걸 얼마나 쉽고 기분 좋게 얻었는가.

> 즉, UI는 UX를 이루는 재료 중 하나예요. UI가 예뻐도 UX가 나쁠 수 있어요.

## 4. UI vs UX 예시로 보기 (평소에 쓰던 영상 앱으로 보면)
- **UI 화면 하나**: 색, 폰트, 버튼 모양, 이미지 배치 — 눈에 보이는 이 화면 자체가 UI (홈 화면, 히어로 배너, TOP10 포스터 레일 목업)
- **UX 흐름 하나**: 앱을 켠다 → 보고싶은 걸 찾는다 → 한 번에 재생 버튼을 누른다
  헤매지 않고, 스트레스 없이, 원하는 걸 빠르게 얻었다면 그 과정 전체에 대한 만족감이 바로 UX예요. UI는 순간의 화면, UX는 시작부터 끝까지의 여정이에요.

## 5. 좋은 UX vs 나쁜 UX
- ✓ **좋은 UX**: 재생 버튼이 바로 보여요 — 뭘 눌러야 할지 한눈에 보이고, 누르면 바로 원하는 일이 일어난다.
- ✕ **나쁜 UX**: 버튼이 어디 있는지 한참 찾아야 해요 — 중요한 버튼이 묻혀 있고, 다음에 뭘 눌러야 할지 계속 고민하게 만든다.

> UI/UX 디자이너는 이 "헤매는 순간"을 미리 찾아서 없애는 사람이에요.

## 6. 서비스 하나가 탄생하기까지 (6단계 개요)
앱 하나가 세상에 나오려면 이 6단계를 거쳐요.
1. **리서치** — 문제와 사용자를 이해하기
2. **기획 · 전략** — 무엇을 만들지 정하기
3. **구조 설계** — 뼈대 그리기 (와이어프레임)
4. **UI 디자인** — 보기 좋게, 브랜드답게
5. **프로토타입 · 테스트** — 직접 써보고 고치기
6. **개발 · 출시** — 개발자와 함께 세상에 내놓기

### Step 1. 리서치
"누구를 위해, 어떤 문제를 풀 것인가"를 알아내는 단계예요. 여기서 방향이 틀어지면 이후 모든 게 흔들려요.
- 사용자 인터뷰 — 실제 사용자를 만나서 불편함과 니즈를 직접 듣기
- 설문 조사 — 많은 사람의 데이터를 모아 패턴 찾기
- 경쟁 서비스 분석 — 비슷한 앱들은 이 문제를 어떻게 풀고 있는지 보기
- 데이터 분석 — 사람들이 앱을 실제로 어떻게 쓰는지 로그로 확인

### Step 2. 기획 · 전략
리서치로 알아낸 걸 바탕으로 "무엇을 만들지, 무엇부터 만들지" 결정해요.
- 문제 정의 — 우리가 진짜로 풀어야 할 문제를 한 문장으로 정리
- 목표 설정 — 이 서비스로 무엇을 이루고 싶은지 정하기
- 우선순위 정하기 — 기획자·개발자·마케터와 함께 뭘 먼저 만들지 논의
- 플로우 스케치 — 사용자가 어떤 순서로 앱을 쓸지 큰 그림 그리기

### Step 3. 정보 구조 & 와이어프레임
색이나 폰트는 아직 신경 안 써요. 화면에 뭐가, 어떤 순서로 있어야 하는지 뼈대만 그려요.
> "이 화면엔 검색이 먼저 와야 할까, 추천 콘텐츠가 먼저 와야 할까?" — 이런 순서와 우선순위를 정하는 게 이 단계의 핵심이에요.

### Step 4. UI 비주얼 디자인
흑백 뼈대에 브랜드의 색, 폰트, 아이콘, 이미지를 입혀 실제 화면처럼 만드는 단계예요.

### Step 5. 프로토타입 & 테스트
화면들을 실제로 눌러볼 수 있게 연결해서, 진짜 사용자에게 써보게 하고 반응을 관찰해요.
- 클릭 가능한 목업 — Figma 등으로 화면들을 실제 앱처럼 연결하기
- 사용성 테스트 — 사람들이 어디서 막히고 헤매는지 직접 관찰
- 피드백 반영 — 문제를 발견하면 다시 고치고, 또 테스트하고 — 반복

### Step 6. 개발 협업 & 출시
디자인을 개발자에게 전달하고, 실제 코드로 구현되는 과정을 함께 확인해요. 출시 후에도 계속 지켜봐요.
- 디자인 핸드오프 — 색상 코드, 간격, 폰트 값을 개발자에게 정확히 전달
- QA (검수) — 실제 구현이 디자인과 똑같은지 하나씩 확인
- 출시 후 모니터링 — 실제 사용자 반응과 지표를 보며 계속 개선하기

## 7. UI/UX 디자이너가 실제로 하는 일
예쁜 화면만 그리는 게 아니에요. 하루를 들여다보면 이런 모습이에요.
- 회의 · 협업 — 기획자·개발자·마케터와 매일 논의하고 조율하기
- 화면 설계 — 와이어프레임과 실제 화면 시안 그리기
- 근거 찾기 — "왜 이렇게 디자인했나요?"에 답할 데이터와 이유 준비
- 디자인 시스템 관리 — 색, 버튼, 아이콘 같은 규칙을 정리하고 지키기
- 사용자 관찰 — 인터뷰, 테스트 영상 보며 사람들의 행동 분석
- 개발 검수 — 실제 구현된 화면이 의도대로 나왔는지 확인

## 8. UI/UX 디자이너 vs 프로덕트 디자이너
- **UI/UX 디자이너** — 화면과 경험에 집중. 주어진 문제를 화면으로 잘 풀어내는 데 집중해요. 와이어프레임, 화면 디자인, 사용성 테스트가 주 업무예요.
- **프로덕트 디자이너** — '무엇을 만들지'부터 관여. 화면 디자인뿐 아니라 어떤 기능을 만들지, 사업적으로 말이 되는지까지 기획자·개발자와 함께 결정해요.

> 회사마다 호칭과 역할 범위가 달라요. 요즘은 두 역할의 경계가 점점 흐려지는 추세예요.

## 9. 그럼 어떤 능력이 필요할까?
**TOOLS · 툴**
- Figma — 화면 디자인, 협업 기본 툴
- 포토샵 · 일러스트레이터 — 이미지·그래픽 작업
- 노션 · 피그잼 — 기획 정리, 협업 문서
→ 툴은 시간이 지나면 바뀌어요. 지금부터 익숙해질 필요는 없어요.

**SOFT SKILLS · 소프트스킬**
- 공감 능력 — 사용자 입장에서 생각하기
- 논리적 설득력 — "왜 이렇게 만들었는지" 설명하기
- 피드백 수용력 — 내 디자인이 비판받아도 괜찮기
→ 이게 진짜 오래가는 힘이에요.

## 10. 전공 · 진로는 이렇게 준비해요
- **전공은 하나가 아니에요** — 시각디자인, 산업디자인은 물론 컴퓨터공학, 심리학을 거쳐 오는 디자이너도 많아요. 중요한 건 전공보다 결과물이에요.
- **포트폴리오가 핵심** — 취업할 때 학점보다 더 많이 보는 건 포트폴리오예요. 실제로 문제를 발견하고 풀어낸 과정을 보여주는 게 중요해요.
- **지금 할 수 있는 것** — 평소 쓰는 앱을 유심히 관찰하기, 공모전 참여하기, 좋아하는 서비스를 직접 그려보며 흉내 내보기.
